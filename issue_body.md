## 背景
UI設計着手前の確認で、画面設計に必要な API 入口が不足していることが判明した。
特に、郵便番号補完、ダッシュボード KPI、CSV 出力は UI から利用する入口が明確でない。
また、API レスポンス形式が混在しているため、UI設計時の停止要因になる。

## 目的
UI利用に必要な API 入口を補完し、あわせてレスポンス形式の統一確認を行う。

## 実装対象ファイル
- `20_api_netshop.gs`
- 必要に応じて `02_triggers.gs`
- 必要に応じて `10_webapp.gs`

## 必須実装
- 郵便番号補完の UI 用 API 入口を追加する
- ダッシュボード KPI の UI 用 API 入口を追加する
- CSV 出力の UI 用 API 入口を追加する
- 必要なら `api_dispatchAction` から呼べるようにする
- 少なくとも今回追加・整理する API について、成功 / 失敗レスポンス形式を揃える
- UI から利用する前提で、戻り値の構造を明確にする

## 確認事項として必ず含めること
- 配列直返し
- `ok + data`
- `error` 文字列
が混在していないか確認すること
- 今回追加対象の API では、UI が扱いやすい統一形式に寄せること

## 業務ルール
- 既存 API 基盤を壊さないこと
- `doPost` は新規追加しない
- 既存の業務ロジックは変えない
- UI 利用の入口整備が目的であり、画面実装は行わない

## 完了条件
- 郵便番号補完 API の UI 用入口が使えること
- ダッシュボード KPI API の UI 用入口が使えること
- CSV 出力 API の UI 用入口が使えること
- 今回対象 API のレスポンス形式が UI 利用前提で統一または整理されていること
- `20_api_netshop.gs` を中心に実コード差分が入ること
- `md/json` だけの変更で終わらないこと

## 非対象
- UI 実装
- 画面遷移設計
- 個人情報表示方針の決定

## 注意
- 調査ではなく実装 / 整理 Issue
- 1 Issue = 1機能群
- 対象外の機能を混ぜないこと
