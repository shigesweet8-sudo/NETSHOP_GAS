## 目的
`20_api_netshop.gs` の更新系 API に残っている固定列番号依存を廃止し、`CONFIG.HEADERS` を基準にヘッダ名から列位置を解決する方式へ統一する。列順が変わっても `統合売上データ` シートで誤更新しない実装に変更する。

## 実装スコープ
- ヘッダ名から列位置を解決する共通処理を追加する
- 更新系 API が固定列番号直書きではなく、共通処理で解決した列位置を使うように変更する
- 少なくとも以下の処理を対象に参照方式を統一する
  - `createItem`
  - `updateItem`
  - `updateItemStatus`
  - `bulkUpdateStatus`
  - `recalculateItemProfit`
  - `bulkRecalculateProfit`
  - `cancelItem`
  - `bulkDisableNetshopRecords`
- 必要に応じて `01_config.gs` の `CONFIG.HEADERS` 参照まわりを補強する
- 既存 API の業務ロジック・入出力・公開スキーマは維持する

## 対象ファイル
- 主対象: `20_api_netshop.gs`
- 必要に応じて: `01_config.gs`

## 制約
- シート名は `統合売上データ` を前提とする
- シートヘッダは `CONFIG.HEADERS` と整合している前提で実装する
- ただし、必要ヘッダが見つからない場合は静かに誤更新せず、明示的に失敗させる
- 既存の読み取り系ロジックや UI は変更しない
- 固定列番号の直書きを更新処理に残さない
- `md/json/txt` のみを変更して完了してはならない。必ず実装ファイルを変更すること

## 非対象
- UI 実装
- API 公開スキーマ変更
- シート構造そのものの変更
- issue整理、設計メモ追加、ドキュメントのみの更新

## 実装ステップ
1. `20_api_netshop.gs` 内の更新系 API で使われている列参照箇所を洗い出し、固定列番号で `getRange` / `setValue` / `setValues` / 行配列アクセスしている部分を特定する。
2. `CONFIG.HEADERS` を基準に、ヘッダ名→1始まり列番号、および必要ならヘッダ名→0始まり配列 index を返す共通関数を `20_api_netshop.gs` に追加する。  
   - 例: シート先頭行から実ヘッダを読み、`CONFIG.HEADERS` と整合確認したうえでマップを返す
   - 必須ヘッダ不足時は例外を投げる
   - 同一実行内で再利用できる形にする
3. `createItem` を修正し、書き込み対象列をヘッダ名で指定して解決した列番号を使って更新する。新規行作成時に行配列を組み立てる場合も、`CONFIG.HEADERS` 順または解決済み index を使って列位置を保証する。
4. `updateItem` / `updateItemStatus` / `bulkUpdateStatus` を修正し、更新対象列をすべてヘッダ解決ベースへ置換する。複数列更新時も固定番号を使わず、列マップから参照する。
5. `recalculateItemProfit` / `bulkRecalculateProfit` を修正し、利益再計算で参照・更新する各列をヘッダ名ベースに統一する。既存の計算式や業務ロジックは変えない。
6. `cancelItem` / `bulkDisableNetshopRecords` を修正し、状態更新・無効化更新・関連列更新をヘッダ解決結果で行うようにする。
7. 可能であれば重複する更新処理を小さな内部ヘルパーに寄せ、今後も固定列番号へ戻りにくい構造に整理する。
8. 対象関数内に固定列番号依存が残っていないことを確認し、必要最低限のログまたはエラーメッセージでヘッダ不整合時に原因が追える状態にする。

## 完了条件
- `20_api_netshop.gs` に実コード差分がある
- 対象更新系 API が固定列番号直書きではなく、ヘッダ名から解決した列位置を使っている
- `統合売上データ` の列順が変わっても、ヘッダが一致している限り誤更新しない
- 必要ヘッダ欠落や不整合時に静かに処理継続せず、明示的に失敗する
- 既存 API の業務ロジックと公開インターフェースを壊していない
- `codex_instruction.md` / `issue_tasks.md` / `openai_response.json` だけの変更では終わっていない

## 検証
- `20_api_netshop.gs` 内で対象関数に固定列番号の直書きが残っていないことを確認する
- `CONFIG.HEADERS` と同じヘッダ順のケースで既存挙動が維持されることを確認する
- `統合売上データ` の列順を入れ替えた想定でも、ヘッダ解決により更新対象列が正しく選ばれることをコード上で確認する
- 必須ヘッダを欠落させた場合に、誤更新ではなく例外や明示エラーになることを確認する
