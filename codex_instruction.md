## 目的
`20_api_netshop.gs` に、複数の管理IDを同一ステータスへ一括更新する `bulkUpdateStatus` API を実装する。既存の API 呼び出し方式に合わせて入口関数を追加し、`updateItemStatus` と整合する更新ルールで、成功件数・失敗件数・成功ID一覧・失敗ID一覧を返すようにする。

## 実装スコープ
- `20_api_netshop.gs` に一括ステータス更新 API 本体を追加する
- 既存の GAS API 入口方式に合わせた呼び出し分岐または公開関数を追加する
- `itemIds`, `status`, `memo` を受け取り、`統合売上データ` シート上の対象行を一括更新する
- `CONFIG.STATUS_LIST` に存在しないステータスは更新せずエラーとして扱う
- 存在しない管理IDは失敗として返す
- 更新時に D列 `DATE` を更新する
- ステータスが `商品登録` で E列 `商品登録日` が空の場合のみ初回設定する
- 返却値に `successCount`, `failureCount`, `successIds`, `failureIds` を含める
- 大量件数でも破綻しにくいよう、対象シートの読み取り・書き込み回数を必要最小限に抑える

## 対象ファイル
- `20_api_netshop.gs`

## 制約
- `doPost` は新規追加しない
- 既存の `updateItemStatus` と矛盾しない判定・更新ルールにすること
- ステータス妥当性判定は `CONFIG.STATUS_LIST` に従い、必要に応じて `CONFIG.STATUS.LISTING` との整合も維持すること
- 対象シートは `統合売上データ`
- 管理IDは B列、ステータスは A列、DATE は D列、商品登録日は E列
- 既存 API 基盤を壊さないこと
- 不明点がある場合でも `20_api_netshop.gs` の既存実装に合わせて最も自然な形で実装すること
- `memo` は入力として受け取るが、既存実装に保存先がない場合は無理に新規列追加しない
- `md/json/txt` のみの変更で終わらせないこと。必ず実装ファイルに実コード差分を入れること

## 非対象
- UI 実装
- `[8] 一括再計算API`
- `[15] キャンセル処理API`
- シート構成変更、新規シート追加
- `doPost` の新設や API 基盤の全面改修
- Issue に記載のない別 API の修正

## 実装ステップ
1. `20_api_netshop.gs` 内の既存 API 入口、`updateItemStatus`、レスポンス形式、シート取得方法、日時の扱い方を確認し、それに揃えて `bulkUpdateStatus` の公開入口を追加する。既存の action/router 方式がある場合はそこに分岐を足す。
2. `bulkUpdateStatus` の入力バリデーションを実装する。
   - `itemIds` が配列であること
   - 空配列を弾くこと
   - `status` が存在すること
   - `status` が `CONFIG.STATUS_LIST` 内の有効値であること
   - 不正な場合は全件更新せず、既存 API 形式に合わせたエラーレスポンスを返す
3. `統合売上データ` シートを一括で読み取り、B列の管理IDから対象行を高速に引けるマップを作る。IDごとに `getRange(...).setValue(...)` を繰り返す実装は避け、可能な限りまとめて読み・まとめて書く構成にする。
4. 対象 `itemIds` を順に処理し、存在する管理IDのみ更新対象にする。
   - A列に新ステータスを設定
   - D列 `DATE` を更新
   - 新ステータスが `商品登録` かつ E列が空の場合のみ E列 `商品登録日` を設定
   - 存在しない管理IDは `failureIds` に追加
5. 書き込みは大量件数でも壊れにくいよう、少なくとも A/D/E 列の対象データを配列で保持してまとめて反映する。既存実装が `setValues` ベースならそれに揃える。行単位の細かい書き込み連打は避ける。
6. 処理結果として、少なくとも以下を返すようにする。
   - `successCount`
   - `failureCount`
   - `successIds`
   - `failureIds`
   必要に応じて既存 API の `ok/error/message` 形式があればそれにも合わせる。
7. `updateItemStatus` と重複するステータス判定や日付更新ルールがある場合は、既存ロジックを再利用するか、少なくとも挙動が一致するように整理する。既存関数を壊さないこと。
8. 必要であれば、既存の単一更新 API と同じレスポンスヘルパーや例外処理に合わせて整形する。

## 完了条件
- `20_api_netshop.gs` に実コード差分が存在する
- 一括ステータス更新 API 本体が追加されている
- 既存の GAS 呼び出し方式に沿った API 入口が追加されている
- 有効なステータス時は複数 ID を同一ステータスへ更新できる
- 不正ステータスは更新せずエラー扱いになる
- 存在しない管理IDは失敗IDとして返る
- `DATE` 更新、および `商品登録` 時の `商品登録日` 初回設定が実装されている
- 返却値に成功件数・失敗件数・成功ID一覧・失敗ID一覧が含まれる
- `md/json/txt` だけの変更ではなく、実装ファイルの変更で完了している

## 検証
- 既存の API 呼び出し方式で `bulkUpdateStatus` を呼べることを確認する
- 例: `itemIds=["CP-A0001","CP-A0002"]`, `status="発送準備中"` で、両IDが存在すれば `successCount=2`, `failureCount=0` になること
- `itemIds` に存在しないIDを混ぜた場合、そのIDだけ `failureIds` に入ること
- `status` に `CONFIG.STATUS_LIST` 外の値を渡した場合、更新せずエラーになること
- `status="商品登録"` で E列が空の行は E列が設定され、既に値がある行は上書きされないこと
- 更新対象行の D列 `DATE` が更新されること
- 既存の `updateItemStatus` の挙動や既存 API 入口を壊していないこと
