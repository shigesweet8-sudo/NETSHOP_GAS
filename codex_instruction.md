## 目的
`20_api_netshop.gs` に、`統合売上データ` を元に SHOP / プラットフォーム別の件数・売上・手数料・送料・仕入・利益を集計して返す API を追加する。既存の月別集計API / ダッシュボードAPI と整合する金額計算を使い、`doPost` は追加しない。

## 実装スコープ
- `20_api_netshop.gs` にプラットフォーム別集計API本体を追加する
- 既存の API 入口に新しいアクション/ルーティングを追加する
- 対象シート `統合売上データ` から F列 `SHOP` を基準に集計する
- SHOP が空の行は除外する
- 返却値として SHOP 単位の以下を含める
  - 件数
  - 売上
  - 手数料
  - 送料
  - 仕入
  - 利益
- 金額系の算出は、同ファイル内または既存集計APIで使っている粗利計算ロジックと整合させる

## 対象ファイル
- 必須: `20_api_netshop.gs`
- 参照候補:
  - `20_api_netshop.gs` 内の既存月別集計API実装
  - `20_api_netshop.gs` 内の既存ダッシュボードAPI実装
  - 同ファイル内の既存レスポンス生成関数 / ルーティング関数 / 共通集計関数
- 変更禁止寄り:
  - `doPost` を新規追加しない
  - Issue対象外ファイルは原則変更しない

## 制約
- 実装は調査メモではなく、実際に動く GAS コードを追加すること
- 既存 API 基盤を壊さないこと
- 既存の `doGet` / ルーティング方式がある場合はそれに乗せること
- 金額系の列参照は、既存の粗利計算・月別集計・ダッシュボードAPIで使っている列定義/計算式を優先して再利用すること
- 列番号やヘッダ名がコード上で一元化されているならそれを使うこと。ハードコードが必要な場合でも既存実装に合わせること
- SHOP は F列基準。SHOP が空の行は必ず無視すること
- 返却キー名は既存 API の命名規則に合わせること
- 不明点:
  - 「プラットフォーム」と「SHOP」の既存用語の使い分けがコード上で異なる可能性があるため、既存APIの命名に合わせること
  - 利益計算式は既存粗利計算と完全一致させること。独自計算を新規定義しないこと

## 非対象
- UI 実装
- `doPost` の追加
- [17] マスター取得API
- 既存 API の全面リファクタ
- md/json/txt のみの変更
- ドキュメントだけを更新して終了すること

## 実装ステップ
1. `20_api_netshop.gs` の既存 API 入口（例: `doGet` 内の action 判定、dispatch 関数など）を確認し、プラットフォーム別集計API用のルートを追加する。既存命名に合わせて `getPlatformSummary` / `getShopSummary` 相当の関数名を採用する
2. 既存の月別集計API / ダッシュボードAPI / 粗利計算ロジックを参照し、`統合売上データ` の対象列（件数、売上、手数料、送料、仕入、利益）を同じ計算根拠で取得できるようにする
3. `統合売上データ` を読み込み、ヘッダ行を除いたデータを走査して、F列 `SHOP` が空でない行のみを対象に SHOP 単位で集計する
4. 集計値として以下を加算する
   - 件数: 対象行数、または既存実装で件数列が定義されている場合はその定義に従う
   - 売上
   - 手数料
   - 送料
   - 仕入
   - 利益
5. 利益は既存粗利計算と同じ算式/取得元を使う。すでに各行に粗利/利益列があるならそれを再利用し、ない場合でも既存APIと同じ導出方法で算出する
6. 返却形式は既存 API のレスポンスラッパーに合わせる。少なくとも、SHOPごとの集計配列またはオブジェクトを JSON で返す
7. 既存処理と競合しないように最小差分で実装し、不要な名前変更や広範囲な整形は避ける

## 完了条件
- `20_api_netshop.gs` に実コード差分がある
- プラットフォーム別集計API本体の関数が追加されている
- 既存 API 入口から新APIを呼べる
- `統合売上データ` をもとに SHOP ごとの件数・売上・手数料・送料・仕入・利益を返せる
- SHOP 空行を除外している
- 利益/金額計算が既存の月別集計API / ダッシュボードAPI / 粗利計算と矛盾しない
- `doPost` を新規追加していない
- `20_api_netshop.gs` 以外を触る場合は必要最小限で、実装ファイルの変更が主体である
- `codex_instruction.md` / `issue_tasks.md` / `openai_response.json` / md/json/txt だけの変更で終わらないこと

## 検証
- 既存の API 入口から新しい action / mode / type を指定したとき、例外なく JSON が返ること
- 返却データに各 SHOP ごとの `件数・売上・手数料・送料・仕入・利益` が含まれること
- SHOP が空の行が集計に含まれないこと
- 既存の月別集計API / ダッシュボードAPI と比較して、同一データセットで金額系ロジックが矛盾しないこと
- 既存APIの action に影響を与えていないこと
- 実際の変更が `20_api_netshop.gs` の実装コードに入っていること
