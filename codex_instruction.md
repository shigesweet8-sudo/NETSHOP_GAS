## 目的
`20_api_netshop.gs` にバリエーション確認APIを実装し、商品登録・更新前に入力内容の妥当性を確認できるようにする。`doPost` は新規追加せず、既存API入口に確認用ルーティングを追加し、エラー一覧・警告一覧を返せる実コードを追加する。

## 実装スコープ
- 入力内容を検証する確認関数の追加
- 既存API入口へのバリエーション確認APIルーティング追加
- 少なくとも以下の確認処理を実装
  - 必須項目の不足
  - ステータス値の妥当性
  - SHOP値の妥当性
  - 金額項目が数値として扱えるか
- 例外で処理を落とさず、確認結果として返却
- 既存の `createItem` / `updateItem` と整合するバリデーション観点に寄せる
- 既存マスター値との整合を保つ

## 対象ファイル
- `20_api_netshop.gs`
- 必要に応じて同ファイル内の既存 `doPost`、APIディスパッチ部、`createItem` / `updateItem` 参照箇所
- もし同API基盤で共通レスポンス生成関数が同ファイル内にある場合はその利用・最小修正を行う

## 制約
- `doPost` は新規追加しない。既存の `doPost` または既存入口に分岐を追加すること
- 実装は `20_api_netshop.gs` の実コード変更を必須とする
- 未入力・`null`・`undefined`・空文字を安全に扱うこと
- 例外送出ではなく、可能な限り `{ errors: [], warnings: [] }` のような確認結果で返すこと
- 既存API基盤を壊さないこと
- Issueに明記のないUI・並び替えAPI・キャンセル処理APIは実装しないこと
- 対象フィールド名や既存マスター取得方法がコードからしか確定できない場合は、`createItem` / `updateItem` で実際に使っている項目名・定数・マスター参照関数を優先して流用すること

## 非対象
- UI実装
- [14] 並び替えAPI
- [15] キャンセル処理API
- 新規の `doPost` 作成
- ドキュメントのみの変更
- Issueに記載のない別API追加やリファクタ大規模化

## 実装ステップ
1. `20_api_netshop.gs` の既存API入口を確認し、バリエーション確認API用のアクション名/分岐を追加する。既存レスポンス形式がある場合はそれに合わせる。
2. `createItem` / `updateItem` の入力前提、必須項目、ステータス判定、SHOP判定、金額項目の扱いを参照し、共通化しすぎず安全に流用できる確認関数を追加する。
3. バリエーション確認API本体関数を追加し、受信payloadを安全に正規化して検証する。例外が起きうる箇所は `try/catch` で吸収し、内部エラーも確認結果またはAPIエラー応答として返す。
4. 検証結果は少なくとも `errors` と `warnings` を含む構造で返す。各要素には判別可能な情報（例: 対象項目、メッセージ、必要ならコード）を持たせる。
5. 必須項目不足チェックを実装する。対象項目は `createItem` / `updateItem` と整合するものを採用し、未入力・空文字・空白のみ文字列を不足として扱う。
6. ステータス値の妥当性チェックを実装する。許容値は既存実装または既存マスター参照に合わせ、未知の値はエラーまたは既存仕様に合わせた警告として返す。
7. SHOP値の妥当性チェックを実装する。既存マスター値や既存判定ロジックに合わせて確認し、不正値を返す。
8. 金額項目について、既存登録更新処理で扱う対象を特定し、数値変換可能かを確認する。空値を許容するかは既存処理準拠とし、非数値はエラーにする。
9. 既存APIへ影響を与えないことを確認し、最低限の動作確認コード/既存テストがあれば更新する。テスト基盤がなければ関数単位で読みやすい実装に留める。

## 完了条件
- `20_api_netshop.gs` に実コード差分がある
- バリエーション確認APIの入口分岐が追加されている
- バリエーション確認API本体関数が追加されている
- 必須項目、ステータス、SHOP、金額項目の検証が実装されている
- `errors` と `warnings` を返せる
- `doPost` を新規追加していない
- 既存の `createItem` / `updateItem` と矛盾する独自仕様を不用意に導入していない
- `codex_instruction.md` / `issue_tasks.md` / `openai_response.json` のみの変更で終わっていない

## 検証
- 既存API入口からバリエーション確認APIを呼べること
- 必須項目欠落時に `errors` に内容が入ること
- 妥当なステータス値は通過し、不正値は検出されること
- 妥当なSHOP値は通過し、不正値は検出されること
- 金額項目に数値文字列を渡した場合は通過し、非数値文字列では検出されること
- 未入力、空文字、`null` 相当入力でも例外で落ちないこと
- 既存の他APIルートの挙動を壊していないこと
