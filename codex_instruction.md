## 目的
`20_api_netshop.gs` に、`統合売上データ` シートの複数レコードを安全に一括無効化する API を追加する。複数IDを受け取り、成功件数・失敗件数・成功ID一覧・失敗ID一覧を返すこと。既存 API 基盤を壊さず、`doPost` は新規追加しない。

## 実装スコープ
- 一括削除 / 無効化 API 本体を `20_api_netshop.gs` に追加する
- 実体は「物理削除ではなく安全な無効化寄り」の処理にする
- API 入口を既存のルーティング方式に合わせて追加する
- 複数の管理IDを受け取り、`統合売上データ` の B列をキーに対象行を判定する
- 存在しないIDは失敗として返す
- レスポンスに以下を含める
  - 成功件数
  - 失敗件数
  - 成功ID一覧
  - 失敗ID一覧
- 既存のレスポンス形式や共通ユーティリティがあればそれに合わせる

## 対象ファイル
- `20_api_netshop.gs`

## 制約
- `doPost` は新規追加しない。既存の API 入口・分岐に追記すること
- 既存データを破壊しないこと
- 物理削除は実装しないこと
- 既存 API 基盤を壊さないこと
- 対象シートは `統合売上データ`
- 管理IDは B列を使用すること
- 無効化方法は、既存コードに無効/削除対象/ステータス系の列運用があればそれを優先利用すること
- 既存の無効化用列が明確でない場合は、既存データ構造を壊さない最小限の安全な方法で「無効化対象」と分かる更新を行うこと
- 不明点がある場合でも、`20_api_netshop.gs` 内で完結する最も安全な実装を優先すること
- `codex_instruction.md` / `issue_tasks.md` / `openai_response.json` のみの変更で終わらせないこと。必ず実装ファイルに実コード差分を入れること

## 非対象
- UI 実装
- 物理削除の強行実装
- 他 Issue の機能追加
- `20_api_netshop.gs` 以外への不要な横展開
- 調査だけで終わる対応

## 実装ステップ
1. `20_api_netshop.gs` の既存 API 入口とルーティング方式を確認し、一括削除 / 無効化API 用の分岐を追加する
2. 既存のレスポンス生成パターンに合わせて、新APIのハンドラ関数を追加する
3. リクエストから複数IDを受け取れるようにする。既存実装に合わせて、配列またはカンマ区切り文字列のどちらか/両方に安全に対応する
4. `統合売上データ` シートを取得し、B列の管理IDを元に対象行を特定する
5. 各IDについて以下を実装する
   - 対象行が見つかる場合: 安全な無効化処理を行う
   - 対象行が見つからない場合: 失敗IDとして記録する
6. 無効化処理は、既存コードに削除対象・無効・キャンセル・状態更新などの近い実装があれば、その更新方式に合わせる
7. 既存の状態更新方式が見当たらない場合は、既存列構成を壊さないことを優先し、既存の空き/末尾列追加のような破壊的変更は避け、既存の管理可能なステータス列がある場合のみ更新する
8. 処理結果として、成功件数・失敗件数・成功ID一覧・失敗ID一覧を返すレスポンスを実装する
9. 例外時も既存 API のエラーハンドリング方針に合わせて安全に失敗レスポンスを返す
10. 関数名や action 名は、Issue の趣旨が分かる命名にする。候補:
   - `bulkDisableNetshopRecords`
   - `apiBulkDisableNetshop`
   - 既存命名規則に合わせた action 文字列

## 完了条件

- 一括削除 / 無効化 API の入口が既存ルーティングに追加されている
- 複数IDを受け取って処理できる
- `統合売上データ` の B列を使って対象判定している
- 物理削除ではなく安全な無効化処理になっている
- 存在しないIDを失敗として返している
- レスポンスに成功件数・失敗件数・成功ID一覧・失敗ID一覧が含まれる
- 既存 API 基盤を壊していない
- `md/json/txt` だけの変更ではなく、実装ファイル変更で完了している

## 検証
- 既存 API 入口から新しい action を指定して呼べることを確認する
- 複数IDを渡したときに、存在するIDは成功、存在しないIDは失敗として集計されることを確認する
- 成功件数と成功ID一覧、失敗件数と失敗ID一覧の整合性が取れていることを確認する
- 同一ID重複入力時の挙動が不自然にならないよう、既存方針に合わせるか、少なくとも二重更新で壊れないことを確認する
- 物理削除が行われていないことを確認する
- 既存の他 API ルートに影響が出ていないことを確認する
