## 目的
Web UI から利用する各種マスター値を返す「マスター取得API」を `20_api_netshop.gs` に実装する。既存の API 基盤に乗せて、ステータス候補・ショップ候補・商品担当者候補・配送元候補・配送業者候補を取得できるようにする。

## 実装スコープ
- `20_api_netshop.gs` にマスター取得用の実関数を追加する
- 既存の API 入口にマスター取得APIのルーティングを追加する
- 少なくとも以下のマスターを返却対象に含める
  - ステータス候補
  - ショップ候補
  - 商品担当者候補
  - 配送元候補
  - 配送業者候補
- 返却形式は UI から使いやすい構造にする
- 未定義マスター指定時は安全に空配列を返す
- `doPost` は新規追加しない

## 対象ファイル
- `20_api_netshop.gs`

## 制約
- 既存コードのハードコード定義と整合すること
- 既存 API の入口・レスポンス形式・共通ユーティリティがある場合はそれに合わせること
- マスター値は新規に独自定義せず、まず `20_api_netshop.gs` 内または既存参照先の定義を再利用すること
- 対応する定義名がコード上で明確でない場合は、既存の集計APIや関連関数で使われている定数・配列・設定取得処理を優先して参照すること
- 情報が不足している場合でも、`20_api_netshop.gs` の既存実装パターンに合わせて最小限で実装すること
- `codex_instruction.md` / `issue_tasks.md` / `openai_response.json` だけを変更して終わらせてはならない

## 非対象
- UI 実装
- [18] バリエーション確認API
- `doPost` の新規追加
- 既存 API 基盤の全面改修
- Issue に記載のない別マスターや別機能の追加

## 実装ステップ
1. `20_api_netshop.gs` の既存 API 入口を確認し、既存の action / path / mode などのルーティング規約に合わせて「マスター取得API」の分岐を追加する
2. 同ファイル内の既存ハードコード定義、定数、候補配列、設定取得ロジックを確認し、以下の候補を返す実装元を特定して再利用する
   - ステータス候補
   - ショップ候補
   - 商品担当者候補
   - 配送元候補
   - 配送業者候補
3. マスター取得本体関数を追加し、UI から扱いやすいように、少なくとも `masters` 配下または同等に種類ごとにアクセスしやすいオブジェクト構造で返す  
   例: 各種候補をキー別配列で持つ構造。配列要素は既存 API 慣例がなければ `{ value, label }` 互換を優先する
4. 特定マスターのみ取得する既存パラメータ慣例がある場合はそれに合わせて対応し、慣例がなければ全件取得を基本とする
5. 未定義のマスター種別が指定された場合でも例外を投げず、該当キーに空配列、または安全な空レスポンスを返すようにする
6. 既存のレスポンスラッパーがある場合は必ずそれを利用し、エラー時も既存 API と整合する形に合わせる
7. 影響範囲を最小化し、既存のプラットフォーム別集計APIなど他 API の分岐を壊していないことを確認する

## 完了条件
- `20_api_netshop.gs` に実コード差分がある
- マスター取得 API の入口分岐が追加されている
- マスター取得 API 本体関数が追加されている
- 少なくとも以下を返せる
  - ステータス候補
  - ショップ候補
  - 商品担当者候補
  - 配送元候補
  - 配送業者候補
- 未定義マスターは空配列で安全に返す
- 既存 API を壊さない
- `md/json/txt` だけの変更で終わっていない

## 検証

- ステータス候補・ショップ候補・商品担当者候補・配送元候補・配送業者候補の各配列が取得できることを確認する
- 未定義のマスター種別指定時に空配列で返ることを確認する
- 他の既存 API ルートが従来通り動作することを確認する
