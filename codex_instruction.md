## 目的
`20_api_netshop.gs` に、複数の管理IDを受け取って「統合売上データ」シート上の粗利を一括再計算する API を追加する。既存の単体粗利再計算ロジックおよび既存の GAS API 呼び出し方式と整合させ、成功件数・失敗件数・成功ID一覧・失敗ID一覧を返すこと。

## 実装スコープ
- 一括再計算 API の入口関数を `20_api_netshop.gs` に追加する
- 既存の GAS の API 入口方式に合わせてルーティング対象を追加する
- 入力 `itemIds: string[]` を受け取り、各管理IDごとに粗利を再計算する
- 対象シートは「統合売上データ」、管理ID は B列、粗利は Q列
- 粗利計算は `決済金額 - サイト利用料 - 配送料 - 仕入れ値`
- 既存の単体粗利再計算 API がある場合、その計算・バリデーション・更新方式と矛盾しないよう共通化または再利用する
- 存在しない管理IDは失敗として返す
- 大量件数でも破綻しにくいよう、必要以上の `getRange/setValue` の繰り返しを避け、可能な範囲でまとめ書きする

## 対象ファイル
- 必須:
  - `20_api_netshop.gs`
- 必要に応じて参照・再利用:
  - 同リポジトリ内の既存単体粗利再計算 API 定義ファイル
  - 既存の API ルーティング/レスポンス整形に関わる `.gs` ファイル
- 候補:
  - `doPost` や API ディスパッチを持つ `.gs` ファイル
  - 粗利計算の共通関数が定義されている `.gs` ファイル

## 制約
- `doPost` は新規追加しない
- 既存 API 基盤を壊さない
- 1 Issue = 1機能として、一括再計算 API 実装のみに限定する
- 実装は調査メモではなく、実コード差分を入れること
- 対象ファイルが完全に明示されているのは `20_api_netshop.gs` のため、まずこのファイルに実装する
- 既存の単体粗利再計算 API 名や補助関数名が不明な場合は、実際のコードベースを確認し、同等責務の既存関数を再利用すること
- 最後を `codex_instruction.md` / `issue_tasks.md` / `openai_response.json` のみの変更で終えないこと

## 非対象

## 実装ステップ
1. `20_api_netshop.gs` の既存 API 一覧とルーティング方式を確認し、同じ形式で bulkRecalculateProfit API の入口を追加する。`doPost` 新設は行わず、既存ディスパッチに分岐を追加する。
2. 既存の単体粗利再計算 API / 粗利計算関数を探索し、計算式・エラーハンドリング・更新方式を再利用できるようにする。単体 API のみ存在し共通化されていない場合は、単体・一括の両方から呼べる内部関数を `20_api_netshop.gs` 内に抽出して整合性を取る。
3. 一括 API 本体を実装する。入力 `itemIds` を配列として検証し、空配列・不正値の扱いは既存 API 流儀に合わせる。
4. 「統合売上データ」シートを一度で読み込み、B列の管理IDから対象行を特定する。存在しないIDは失敗IDに加える。
5. 対象行ごとに必要列から粗利を算出し、Q列の更新値を準備する。既存単体 API が参照している列定義がある場合はそれに合わせること。計算式は `決済金額 - サイト利用料 - 配送料 - 仕入れ値` を満たすこと。
6. 書き込みは可能な限りまとめて行う。行が離散している場合でも、既存実装方針に従い、少なくともシート再読込や1件ずつの不要な書き込みを避ける。更新成功した ID を success 側に追加する。
7. API レスポンスとして、少なくとも以下を返すようにする:
   - `successCount`
   - `failureCount`
   - `successItemIds`
   - `failureItemIds`
   必要に応じて既存レスポンス形式に合わせて `status` や `message` を含める。
8. 既存単体 API と矛盾しないことを確認し、単体 API の動作を壊していないかを確認する。必要なら軽微な内部関数整理を実施する。

## 完了条件
- `20_api_netshop.gs` に一括再計算 API の実コード差分がある
- 既存の GAS API 呼び出し方式に従った入口が追加されている
- 複数の管理IDを受けて粗利再計算が実行できる
- 存在しない管理IDを失敗として返す
- レスポンスに `successCount` / `failureCount` / `successItemIds` / `failureItemIds` が含まれる
- 既存の単体粗利再計算 API と計算ロジックが矛盾しない
- 実装変更は real code files を含み、`md/json/txt` だけで終わっていない

## 検証
- 既存 API 呼び出し形式で bulkRecalculateProfit 相当の action/function 名を指定して呼び出せること
- 入力例 `{"itemIds":["CP-A0001","CP-A0002","DN-A0003"]}` で処理が走ること
- 存在するIDのみ成功一覧に入り、存在しないIDは失敗一覧に入ること
- 粗利列 Q が期待通り更新されること
- 単体粗利再計算 API と同じデータに対して同じ粗利結果になること
- 大量件数でも、全件ループ内で不要なシートI/Oを繰り返さない実装になっていること
