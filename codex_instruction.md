## 目的
`20_api_netshop.gs` 内の UI 利用対象 API について、成功 / 失敗レスポンス形式を一貫した構造に統一する。最低限 `ok`, `error`, `data` を共通化し、配列直返し・文字列直返し・オブジェクト返しの混在を解消する。既存 UI / 呼び出し側への破壊的変更は避け、必要に応じて互換維持のための移行的なフィールドを残す。

## 実装スコープ
- `20_api_netshop.gs` に共通レスポンス生成処理を追加する
- 以下の API の戻り値を共通レスポンス形式へ統一する
  - `listItems`
  - `getItem`
  - `createItem`
  - `updateItem`
  - `updateItemStatus`
  - `bulkUpdateStatus`
  - `recalculateItemProfit`
  - `bulkRecalculateProfit`
  - `getMonthlySummary`
  - `getPlatformSummary`
  - `getDashboardSummary`
  - `getMasters`
  - `validateVariation`
  - `exportCsv`
  - 郵便番号補完 API
  - disable / archive / cancel 系 API
- 既存コードの `try/catch` やエラーハンドリングがある場合は、失敗時に `ok: false` と `error` を返すよう統一する
- 主結果は原則 `data` に格納する
- 互換維持が必要な場合のみ、既存で参照されうる主キーを補助的に残す。ただし新しい標準は `data` を正とする

## 対象ファイル
- 必須: `20_api_netshop.gs`
- 追加候補: なし（本 Issue の実装は原則このファイル内で完結させる）

## 制約
- 実装は調査メモではなく、実際のコード変更にすること
- UI 実装やシート構造変更は行わないこと
- API 基盤を壊さないこと。既存呼び出し側に破壊的変更になりそうな箇所は互換維持を優先すること
- シート内部仕様・内部カラム名・内部表現を新たにレスポンスへ露出しないこと
- 失敗レスポンスの `error` は UI 共通ハンドラで扱いやすい形にそろえること。少なくとも文字列ではなく、一定のオブジェクト構造にすること
- 不明点:
  - disable / archive / cancel 系 API と郵便番号補完 API の正確な関数名はファイル内実装から特定すること
  - 既存互換が必要な戻り値キーは、実装中に現行の返却形式を見て最小限にとどめること

## 非対象
- UI 側の受け取り処理変更
- API 一覧外の機能追加
- シート構造変更
- ドキュメントのみの更新
- 返却形式統一と無関係なリファクタリング

## 実装ステップ
1. `20_api_netshop.gs` を確認し、対象 API とそれらが返している現行フォーマットを把握する
2. ファイル内に共通レスポンスヘルパーを追加する。例:
   - 成功: `ok: true`, `data`, `error: null`
   - 失敗: `ok: false`, `data: null`, `error: { message, code? }`
   必要なら互換維持のために `data` の内容をトップレベルへ浅く複製する補助オプションを用意してよいが、乱用しないこと
3. 各 API の正常系 return をヘルパー経由へ変更する
   - 配列を返していた関数は `data` に配列を入れる
   - 文字列を返していた関数は `data` にオブジェクト化して入れるか、意味のある主結果オブジェクトへ包む
   - 既にオブジェクト返しの関数も `ok/error/data` を付与して統一する
4. 各 API の異常系を統一する
   - `try/catch` がある場合は catch で共通エラーレスポンスを返す
   - バリデーション失敗や業務エラーも同じ構造に寄せる
   - 既存で `throw` 前提の箇所は、当該 API の返却仕様を崩さない範囲で `ok: false` 応答へ寄せる
5. サマリー系・CSV系・郵便番号補完・disable / archive / cancel 系 API も同様に統一する
   - CSV エクスポート結果は、UI が扱うメタ情報を `data` に格納する
   - 生文字列や生配列の返却を残さない
6. 既存呼び出し互換に配慮し、破壊的になりやすい API に限り、必要最小限の移行的フィールドを残す
   - 例: 既存で `items` を直接返していたなら `data.items` を正としつつ、必要最小限でトップレベル `items` を残す
   - ただし API ごとにバラバラな形式を温存しないこと
7. ファイル全体で対象 API の返却が統一されているか確認し、不要な直返しを解消する

## 完了条件
- `20_api_netshop.gs` に実コード差分がある
- 対象 API の成功時レスポンスが原則 `ok: true`, `error: null`, `data: ...` に統一されている
- 対象 API の失敗時レスポンスが原則 `ok: false`, `data: null`, `error: {...}` に統一されている
- 配列直返し・文字列直返し・生オブジェクト直返しが、対象 API では解消されている
- 既存 UI / 呼び出し側への破壊的変更を避けるための互換維持が必要な箇所には最小限の配慮が入っている
- `codex_instruction.md` / `issue_tasks.md` / `openai_response.json` のみを変更して終了しないこと。必ず `20_api_netshop.gs` の実装を変更すること

## 検証
- `20_api_netshop.gs` 内で対象 API の return 文を確認し、共通レスポンスヘルパー経由へ置き換わっていることを確認する
- 正常系:
  - 一覧取得系は `ok === true` かつ `data` に配列または結果オブジェクトが入る
  - 詳細取得 / 作成 / 更新系は `ok === true` かつ `data` に主結果が入る
  - サマリー / マスタ / 郵便番号補完 / CSV 系も同一の包み方になっている
- 異常系:
  - バリデーション失敗、未存在、更新失敗などで `ok === false` と `error.message` が返る
  - `error` が文字列や生の例外オブジェクトのまま返っていない
- 互換性:
  - 既存で広く参照されそうなトップレベルキーを残した場合でも、標準の参照先が `data` にそろっていることを確認する
