## 目的
`20_api_netshop.gs` に、指定した管理IDの商品/取引を `キャンセル` ステータスへ更新するキャンセル処理APIを実装する。既存の API 入口に統合し、`doPost` の新規追加は行わない。

## 実装スコープ
- `統合売上データ` シートを対象に、管理ID（B列）で対象行を特定するキャンセル処理関数を追加する
- 対象行のステータス（A列）を `キャンセル` に更新する
- 必要に応じて DATE（D列）を更新する
- 既存 API 基盤の入口に、キャンセル処理APIの分岐を追加する
- 存在しないID指定時は安全にエラーを返す
- 既存のステータス更新ロジックと矛盾しないように既存関数/共通処理があればそれを優先的に流用する

## 対象ファイル
- `20_api_netshop.gs`

## 制約
- `doPost` は新規追加しない
- 変更は実装ファイルに対して行うこと
- 既存 API のルーティング方式・レスポンス形式・エラーハンドリング方針に合わせること
- 対象シートは `統合売上データ`
- 管理ID は B列、ステータスは A列、DATE は D列
- 既存コードから関数名や action 名が明確でない場合は、既存の命名/分岐規約に合わせて最小差分で実装する
- 不確実性:
  - API入口の action / method / parameter 名は Issue に明記されていないため、`20_api_netshop.gs` の既存分岐に合わせて推定して実装する
  - DATE 更新条件が厳密には未定義のため、既存ステータス更新処理に DATE 更新ルールがある場合はそれに合わせ、なければキャンセル更新時に現在日時を書き込む

## 非対象
- UI 実装
- [16] 一括削除 / 無効化API
- `doPost` の新設や API 基盤全体のリファクタリング
- md/json/txt のみの変更

## 実装ステップ
1. `20_api_netshop.gs` 内の既存 API 入口（`doPost` もしくは既存の action 分岐処理）を確認し、他APIと同じ形式でキャンセル処理用の分岐を追加する
2. 既存にステータス更新用の共通関数があれば流用し、なければ管理IDを受け取って `統合売上データ` を検索するキャンセル処理関数を追加する
3. シート `統合売上データ` の全データまたは必要範囲を読み取り、B列の管理ID一致行を特定する
4. 一致行が存在しない場合は、既存APIのエラー形式に合わせて安全なエラーレスポンスを返す
5. 一致行が存在する場合は、A列を `キャンセル` に更新する
6. 既存の更新仕様に合わせて D列の DATE を更新する。既存ルールが見当たらない場合は `new Date()` を設定する
7. 正常時は既存APIと整合する成功レスポンスを返す
8. 既存の並び替えAPIなど他分岐を壊していないことを確認し、最小差分でまとめる

## 完了条件
- `20_api_netshop.gs` に実コード差分がある
- キャンセル処理API本体が追加されている
- 既存 API 入口にキャンセル処理の分岐が追加されている
- 指定IDのステータスを `キャンセル` に更新できる
- 存在しないIDで安全にエラーを返せる
- `md/json` だけの変更で終わっていない
- `doPost` の新規追加をしていない

## 検証
- 既存 API 入口からキャンセル用 action を指定して呼び出し、対象管理IDの A列が `キャンセル` に更新されることを確認する
- 対象管理IDの D列 DATE が既存仕様どおり更新されることを確認する
- 存在しない管理IDを指定した場合に、例外で API 全体が壊れずエラーレスポンスが返ることを確認する
- 既存の他API分岐に影響が出ていないことを確認する
- 最終差分に `20_api_netshop.gs` の実装変更が含まれることを確認する
- `codex_instruction.md` / `issue_tasks.md` / `openai_response.json` のみを変更して終了しないこと
