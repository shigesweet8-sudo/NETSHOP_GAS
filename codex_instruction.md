## 目的
UI設計で利用するためのAPI入口を追加・整理する。`20_api_netshop.gs` を中心に、郵便番号補完・ダッシュボードKPI・CSV出力の UI 用 API 入口を実装し、今回対象APIの成功/失敗レスポンス形式を UI が扱いやすい形に統一する。

## 実装スコープ
- 郵便番号補完の UI 用 API 入口を追加
- ダッシュボード KPI の UI 用 API 入口を追加
- CSV 出力の UI 用 API 入口を追加
- 必要なら `api_dispatchAction` から各入口を呼べるように接続
- 今回追加・整理する API について、レスポンス形式の混在を解消
- UI利用前提で戻り値構造を明確化
- 既存業務ロジックは変更せず、既存API基盤を壊さない範囲でラッパー/入口整備を行う

## 対象ファイル
- 主対象: `20_api_netshop.gs`
- 必要に応じて: `10_webapp.gs`
- 必要に応じて: `02_triggers.gs`
- 既存実装参照候補:
  - `20_api_netshop.gs` 内の既存API関数
  - `api_dispatchAction` 定義箇所
  - 既存のレスポンス生成ヘルパーがあればその定義箇所

## 制約
- 実装は調査ではなく、実コード変更を行うこと
- `doPost` は新規追加しない
- 既存の業務ロジックは変えない。既存処理がある場合は UI 用入口から呼び出す構成を優先する
- 既存 API 基盤を壊さないこと。既存利用者に影響する既存レスポンス変更は避け、今回対象の UI 用入口で統一する
- 少なくとも今回追加対象APIでは、`配列直返し` / `ok + data` / `error` 文字列 の混在を避けること
- 統一形式は UI が扱いやすい JSON オブジェクトにすること。例: 成功時 `{ ok: true, data: ... }`、失敗時 `{ ok: false, error: { message: "..." } }`
- 変更は `md/json/txt` のみで終わらせないこと
- 情報不足がある場合でも、既存コードから最も自然な命名・接続方法を推定して実装すること

## 非対象
- UI 実装
- 画面遷移設計
- 個人情報表示方針の決定
- 既存業務ロジック自体の仕様変更
- 今回対象外APIの全面リファクタリング

## 実装ステップ
1. `20_api_netshop.gs` を確認し、既存の API 入口、`api_dispatchAction`、郵便番号補完/ダッシュボードKPI/CSV出力に相当する既存業務処理や関連関数を特定する。
2. 今回対象API専用のレスポンス統一ヘルパーが未整備なら `20_api_netshop.gs` に追加する。成功/失敗の返却形式を明示し、UIで扱えるオブジェクト構造に統一する。
3. 郵便番号補完の UI 用 API 入口を追加する。既存の業務関数がある場合はその結果をラップして返し、未入力・不正入力・処理失敗時も統一形式で返す。
4. ダッシュボード KPI の UI 用 API 入口を追加する。既存集計ロジックがある場合は流用し、戻り値を統一形式に整える。
5. CSV 出力の UI 用 API 入口を追加する。既存CSV生成処理がある場合は流用し、UI側が解釈できる戻り値構造に整える。文字列直返しや配列直返しを避け、必要なメタ情報があれば `data` 配下に含める。
6. 必要に応じて `api_dispatchAction` へ今回追加した各 UI 用 action を登録し、既存ルーティングから呼び出せるようにする。
7. 既存の今回対象処理に `配列直返し` / `error` 文字列返却 / 形式不統一がある場合、既存業務ロジックを変えずに UI 用入口側で吸収・整形する。
8. 例外処理を追加し、エラー時に文字列や配列をそのまま返さず、統一した失敗レスポンスを返すようにする。
9. 可能なら既存のテスト/確認コードに合わせて、最低限の動作確認用呼び出しパスが壊れていないことを確認する。

## 完了条件
- `20_api_netshop.gs` を中心に実コード差分がある
- 郵便番号補完 API の UI 用入口が追加され、呼び出し可能である
- ダッシュボード KPI API の UI 用入口が追加され、呼び出し可能である
- CSV 出力 API の UI 用入口が追加され、呼び出し可能である
- 必要なら `api_dispatchAction` 経由で各APIが利用可能になっている
- 今回追加・整理したAPIが、成功時/失敗時ともに UI 向けの統一レスポンス形式を返す
- 少なくとも今回対象APIで、`配列直返し` / `ok + data` / `error` 文字列 の混在が解消されている
- 既存業務ロジックは変更せず、入口整備で対応している
- `codex_instruction.md` / `issue_tasks.md` / `openai_response.json` だけの変更では終わっていない

## 検証
- 各 action 名で `api_dispatchAction` から呼び出せることを確認する
- 郵便番号補完: 正常入力時に `{ ok: true, data: ... }`、異常系で `{ ok: false, error: { message: ... } }` になることを確認する
- ダッシュボード KPI: 正常時に UI が参照しやすいオブジェクト構造で `data` が返ることを確認する
- CSV 出力: UI 用入口の戻り値が文字列直返しではなく、統一形式の `data` 配下に格納されていることを確認する
- 今回対象APIについて、配列直返し・単純文字列エラー返しが残っていないことを確認する
- 既存の他APIや `doPost` 追加が発生していないことを確認する
